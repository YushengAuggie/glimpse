What it does

A small service that turns uploaded images into resized thumbnails. Users upload an image; the service stores the original, generates several sizes asynchronously, and serves them from a CDN.

Design goal: keep the upload path fast by doing the heavy resizing work off the request, with retries and a clear record of each job's status.

Storage & CDN

Stateless HTTP service. Validates the file, streams the original to object storage, writes a jobs row (status=pending), enqueues a resize job, and returns immediately with a job id.

Workers pull jobs from the queue, generate each configured size, write them to storage, and flip the job to done (or failed after N retries). Scale workers on queue depth.

Originals and thumbnails live in one bucket under predictable keys (img/{id}/{size}.webp). The CDN caches by key; cache busting is just a new id, so we never overwrite.

Operational notes

How are failures handled?
Jobs retry with backoff up to 3 times; after that they're marked failed and surfaced on a small ops dashboard. The original is always kept, so a failed job can be re-queued by hand.
What about duplicate uploads?
The upload API hashes the file and reuses an existing id if the same bytes were already processed — cheap dedupe that avoids redundant resize work.
Where are the bottlenecks?
Resize CPU on the workers. It scales horizontally on queue depth; the API and storage are well within headroom at current traffic.
